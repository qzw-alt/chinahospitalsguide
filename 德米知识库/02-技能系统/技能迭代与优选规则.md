## 🎯 核心原则

### 原则1：永远只保留最优解

```
新技能出现 → 与现有技能对比 → 优胜劣汰 → 只保留更好的
     ↑                                              ↓
     └────────────── 持续迭代 ←─────────────────────┘
```

**禁止行为：**
- ❌ 同时保留功能重复的多个技能
- ❌ "先留着看看"的拖延
- ❌ 因为"可能有用"而保留

**强制行为：**
- ✅ 功能重叠时立即对比测试
- ✅ 明确淘汰较差的技能
- ✅ 只保留真正好用的

## 📋 技能对比流程

### 步骤1：发现新技能

当发现新技能时，先问自己：
1. **这个技能解决什么问题？**
2. **我现有技能中有没有解决同样问题的？**
3. **如果有 → 进入对比流程**
4. **如果没有 → 评估是否需要**

### 步骤2：对比维度

| 维度 | 权重 | 评估标准 |
|------|------|---------|
| **稳定性** | 40% | 是否经常报错/崩溃 |
| **易用性** | 30% | 参数是否简洁、文档是否清晰 |
| **功能性** | 20% | 功能是否更强大/更全面 |
| **速度** | 10% | 响应速度、执行效率 |

## 🧠 肌肉记忆口诀

### 口诀1：一山不容二虎
```
同类型技能只能有一个
新技能出现 → 必有一战 → 胜者留下
```

### 口诀2：用进废退
```
3天不用 → 质疑必要性
7天不用 → 考虑淘汰
30天不用 → 必须淘汰
```

### 口诀3：宁缺毋滥
```
10个普通技能 < 1个精通技能
技能贵精不贵多
```

## 💡 实战案例

### 案例：搜索技能迭代

**初始状态：**
- web-search（基础搜索）

**第一次迭代：**
- 发现：multi-search-engine（多引擎搜索）
- 对比：multi-search-engine > web-search
- 决策：保留 multi-search-engine，淘汰 web-search
- 结果：搜索能力大幅提升

**第二次迭代：**
- 发现：tavily-search（深度搜索）
- 对比：与 multi-search-engine 互补，非替代关系
- 决策：两者都保留，形成组合
- 结果：广度+深度 = 完美搜索组合

**第三次迭代（假设）：**
- 发现：new-search-skill（新一代搜索）
- 对比：比 multi-search-engine + tavily 更好
- 决策：安装 new-search-skill，淘汰前两者
- 结果：持续优化

## ⚠️ 禁止事项

### 绝对禁止

1. **囤积技能**
   - ❌ "这个可能以后有用"
   - ❌ "先留着吧"

2. **拖延淘汰**
   - ❌ "下周再删"
   - ❌ "等有空再说"

3. **重复安装**
   - ❌ 安装功能相同的多个技能
   - ❌ "试试这个好不好用"

4. **忘记技能**
   - ❌ 安装了却不用
   - ❌ "满身是刀，一把不用"
